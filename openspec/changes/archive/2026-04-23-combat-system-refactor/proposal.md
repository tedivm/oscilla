## Why

The engine currently hardcodes `strength`, `dexterity`, and `hp` as if they were universal combat stats — they are not. Any game that does not define those exact stat names gets silently broken combat with no error at load time. This is a correctness bug, not just a flexibility gap.

The combat loop in `oscilla/engine/steps/combat.py` reads `player.stats["strength"]` and `player.stats["dexterity"]` directly, and the `EnemySpec` model carries `hp`, `attack`, and `defense` as fixed-schema fields. A game using `power` and `vitality` instead of `strength` and `hp` cannot use the combat system at all today.

This change removes all stat assumptions from the engine. Combat resolution becomes a manifest-defined behavior that each content package owns. The engine dispatches to whatever `CombatSystem` a game declares and has no opinion about the specifics.

The word "combat" is used loosely throughout. A `CombatSystem` governs any two-party adversarial encounter — a sword fight, an arm-wrestling match, a courtroom trial, a chess game, a negotiation. Stats, defeat conditions, damage formulas, and available actions are all author-defined; the engine imposes no physical-violence semantics.

## What Changes

- Introduce a `CombatSystem` manifest kind (`kind: CombatSystem`) that governs any two-party adversarial encounter — a sword fight, a courtroom trial, an arm-wrestling match, a chess game. The system defines defeat conditions, damage formula lists, player turn mode, turn order, initiative formulas, skill context strings, and system-provided skills. "Combat" is intentionally a loose term; the engine imposes no physical-violence semantics. Stats, victory conditions, and available actions are fully author-defined. Multiple combat systems per game are supported; individual `CombatStep`s can name a specific system or fall back to the game default. A per-step `combat_overrides: CombatStepOverrides` field allows any `CombatSystemSpec` field to be overridden for a single encounter without authoring a full separate manifest.
- Remove all hardcoded stat references (`strength`, `dexterity`, `hp`) from the engine combat layer.
- Remove the fixed `hp`, `attack`, `defense`, and `xp_reward` fields from `EnemySpec`. Replace with `stats: Dict[str, int]` (fully author-defined) and `on_defeat_effects: List[Effect]` (for XP grants, stat changes, milestone grants, or any other reward).
- Extract the existing hardcoded combat logic into a `CombatSystem` manifest in the testlandia content package — the reference implementation that authors can adopt, adapt, or ignore.
- Replace hardcoded vital-threshold defeat checks with `player_defeat_condition` and `enemy_defeat_condition` — full `Condition` trees evaluated each round using the existing condition system. A new `enemy_stat` condition leaf is added to the `Condition` union, allowing defeat conditions that check arbitrary keys in `enemy_stats`. This leaf evaluates to `false` with a warning when called outside a combat context.
- Damage formulas are lists of `DamageFormulaEntry` objects (`player_damage_formulas`, `enemy_damage_formulas`). Each entry names a `target_stat` (the key decremented), a `formula` (Jinja2 int expression), and an optional `display` label shown in the combat HUD. All entries in a list apply in order before the defeat condition is evaluated.
- Support `player_turn_mode: "choice"` on `CombatSystem` — the player selects a combat skill or combat item each round; no auto-attack fires. `player_damage_formulas` must be absent in this mode (hard load-time error if both are present). Add `combat_damage_formulas: List[DamageFormulaEntry]` to `SkillSpec` for per-move damage in choice mode.
- Relax `SkillSpec.contexts` from a fixed enum to arbitrary strings. Each `CombatSystem` declares `skill_contexts: List[str]`; a skill or item is eligible in a combat system when its `contexts` list intersects with the system's `skill_contexts`. `"overworld"` remains a reserved built-in context.
- Add `ItemSpec.contexts: List[str]`, `ItemSpec.combat_damage_formulas: List[DamageFormulaEntry]`, and `EquipSpec.combat_damage_formulas: List[DamageFormulaEntry]`. Context-scoped items appear as choice-mode actions; equip formulas fire automatically each round in auto mode.
- Add `turn_order: "player_first" | "enemy_first" | "initiative"` to `CombatSystemSpec`. When `"initiative"`, both `player_initiative_formula` and `enemy_initiative_formula` are required; they are rendered in `CombatFormulaContext` each round — higher result acts first. `initiative_tie: "player_first" | "enemy_first" | "random"` resolves equal rolls. Defeat check after the first actor's phase short-circuits the round.
- Require each `game.yaml` to have a resolvable combat system before any adventure with a `combat` step can execute. If a game has exactly one registered `CombatSystem`, it becomes the default automatically. If none is registered and a `combat` step is encountered, a hard load-time error is raised.
- Widen `step_state` from `Dict[str, scalar]` to `Dict[str, Any]` in-memory (the database column is already `JSON` / `Any`). This allows the combat loop to persist the full `enemy_stats` dict as a single key.
- Fix the pre-existing bug where `stat_change target='enemy'` ignored the `stat` field and always mutated the enemy's vital stat. After this change it correctly mutates the named key in `enemy_stats`.
- Deprecate `heal target='enemy'` with a load-time warning. Authors should use `stat_change target='enemy'` instead. The `HealEffect` player path is unchanged and out of scope.
- Extract enemy action dispatch into a dedicated `enemy_action_phase()` function — the single hook through which the future Decision Tree AI change can slot alternative enemy behavior without restructuring the combat loop.
- Update the combat template context: `combat.enemy_hp` (scalar) becomes `combat.enemy_stats` (dict). This is a breaking change for any content using `{{ combat.enemy_hp }}` — all such templates must migrate to `{{ combat.enemy_stats['hp'] }}` (or whichever vital stat the game uses).

## Capabilities

### New Capabilities

- `combat-system`: A `CombatSystem` manifest kind that governs any two-party adversarial encounter (sword fight, courtroom trial, arm wrestling, chess — the engine imposes no physical-violence semantics). Declares `player_defeat_condition`, `enemy_defeat_condition`, `player_damage_formulas`, `enemy_damage_formulas` (each a list of `DamageFormulaEntry`), `player_turn_mode` (`"auto"` or `"choice"`), `turn_order` (`"player_first"`, `"enemy_first"`, or `"initiative"`), optional initiative formulas, `initiative_tie`, `skill_contexts`, and `system_skills: List[SystemSkillEntry]` — skills present in the choice-mode menu for that encounter type; each entry carries an optional player-state `condition` that hides the skill when it evaluates to `False` that round. `on_combat_start: List[Effect]` fires once when a new combat begins (not on resume); `on_combat_end: List[Effect]` fires once when combat resolves regardless of outcome; `on_combat_victory: List[Effect]` fires after `on_combat_end` on a win; `on_combat_defeat: List[Effect]` fires after `on_combat_end` on a loss; `on_round_end: List[Effect]` fires at the end of each complete round if no defeat occurred. `combat_stats: List[CombatStatEntry]` declares ephemeral integer stats initialized at encounter start and discarded at encounter end — never written to global player stats; accessible as `combat_stats['name']` in all formula expressions; mutable via `stat_change target='combat'` or `stat_set target='combat'` effects in any lifecycle hook. All lifecycle hooks and `combat_stats` default to empty. Formula context (`CombatFormulaContext`) includes `player` (effective stats), `enemy_stats` (dict), `combat_stats` (dict), and `turn_number`. `DamageFormulaEntry` gains an optional `target: Literal["player", "enemy", "combat"] | None` field to route formula damage to the appropriate stat namespace. The engine dispatches all combat arithmetic to the declared system; no stat assumptions remain in the engine. Multiple systems per game supported; a single registered system is auto-promoted to default. A missing system when a `combat` step is present is a hard load-time error.
- `enemy-stat-condition`: A new `enemy_stat` leaf in the `Condition` union. Checks a named key in `enemy_stats` using the standard comparator fields (`gte`, `lte`, `gt`, `lt`, `eq`). Valid only inside `CombatSystem` defeat conditions; evaluates to `false` with a warning outside combat context. A parallel `combat_stat` leaf (`CombatStatCondition`) checks `combat_stats` values; valid in defeat conditions and `system_skills` conditions (evaluated during combat rounds); evaluates to `false` with a warning outside combat context. Both compose freely with `all`, `any`, and `not`.

### Modified Capabilities

- `manifest-system`: New `CombatSystem` manifest kind and `CombatStepOverrides` model registered; `EnemySpec` schema breaking change (`stats` dict replaces `hp`/`attack`/`defense`/`xp_reward`; `on_defeat_effects` added); `CombatStep` gains optional `combat_system: str | None` and `combat_overrides: CombatStepOverrides | None`; `GameSpec` gains `default_combat_system: str | None`; `SkillSpec.contexts` relaxed from fixed enum to `List[str]`; `SkillSpec` gains `combat_damage_formulas: List[DamageFormulaEntry]`; `ItemSpec` gains `contexts: List[str]` and `combat_damage_formulas: List[DamageFormulaEntry]`; `EquipSpec` gains `combat_damage_formulas: List[DamageFormulaEntry]`; `DamageFormulaEntry` gains `target: Literal["player", "enemy", "combat"] | None` to route damage to the appropriate stat namespace; `CombatStatEntry` is a new model (`name: str`, `default: int = 0`) for declaring ephemeral combat-internal stats; `CombatSystemSpec` gains `system_skills: List[SystemSkillEntry]`, `on_combat_start: List[Effect]`, `on_combat_end: List[Effect]`, `on_combat_victory: List[Effect]`, `on_combat_defeat: List[Effect]`, `on_round_end: List[Effect]`, and `combat_stats: List[CombatStatEntry]`; `CombatStepOverrides` gains the same fields; `SystemSkillEntry` is a new model with `skill: str` and `condition: Condition | None = None`.
- `adventure-pipeline`: Combat step dispatch reads defeat conditions and formula lists from the resolved `CombatSystem` manifest merged with any `combat_overrides`. `resolve_turn_order()` determines acting order each round (including initiative rolls). Defeat check after the first actor short-circuits the round. `player_action_phase()` dispatches based on `player_turn_mode` (auto-formulas or choice menu of system skills + player-owned skills/items + always-present "Do Nothing" fallback); `enemy_action_phase()` extracted as forward-compat hook. `on_combat_start` fires once at encounter start (skip on resume); `on_combat_end` fires once at resolution before `on_defeat_effects`, loot, and branch dispatch; `on_round_end` fires at the end of each complete round if no defeat occurred. `combat_stats` initialized from declared defaults at encounter start (loaded from `step_state["combat_stats"]` on resume). Formula dispatch uses `resolve_target(entry.target, ...)` to route damage to `enemy_stats`, `player.stats`, or `combat_stats` based on `DamageFormulaEntry.target`. `stat_change target='enemy'` correctly targets named stats. `heal target='enemy'` emits a deprecation warning.
- `player-state`: `step_state` type widened to `Dict[str, Any]` in-memory. Enemy stat dict persisted under `step_state["enemy_stats"]` for save/resume across combat rounds. Combat-internal stat dict persisted under `step_state["combat_stats"]`; discarded (not written to player stats) when the encounter resolves.

## Impact

- **Breaking**: `EnemySpec` YAML schema changes — all existing enemy manifests must be migrated (`hp`/`attack`/`defense`/`xp_reward` removed; `stats` and `on_defeat_effects` added).
- **Breaking**: `combat.enemy_hp` removed from template context — templates using it must migrate to `combat.enemy_stats['<vital_stat>']`.
- **Breaking**: Games with `combat` steps must declare at least one `CombatSystem` manifest.
- `oscilla/engine/models/enemy.py` — `EnemySpec` fields replaced
